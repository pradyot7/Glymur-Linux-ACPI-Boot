Glymur ACPI Overview:

Platform context: Qualcomm SM8480 (Glymur), ARM64, HW-Reduced ACPI (no legacy IO).
Applicable to any ARM64 ACPI platform; Qualcomm-specific notes are called out.

1. What Is ACPI?

ACPI (Advanced Configuration and Power Interface) is an open standard that defines
how firmware describes hardware to the operating system. It covers:

  • Hardware topology description — what devices exist, where they are (MMIO base,
    IRQ, DMA)
  • Power management — device power states (D0/D3), system sleep states (S0/S5),
    CPU idle (C-states), CPU frequency (P-states)
  • Configuration — boot-time device parameters, shared resource description

Core Design Principle

Firmware initializes hardware. OS reads ACPI tables to discover what firmware left
running.

This is the fundamental difference from Device Tree:

Aspect Device Tree ACPI
Philosophy Describe hardware; kernel owns init Firmware inits hardware; OS observes
Clock management Kernel calls clk_set_rate() Firmware set clocks; driver uses fixed_clk_rate
Bus init Kernel calls icc_set_bw() via DT nodes Skipped or handled via SCMI
Update mechanism DTB blob updated in firmware image ACPI tables loaded from partition at boot
Namespace DT compatible strings ACPI _HID / _CID strings

2. ACPI Table Architecture

2.1 Table Hierarchy

On Qualcomm platforms, ACPI tables are compiled into acpi.lzma and stored in
the BOOT_FW2 partition. In uefi AcpiPlatformDxe loads .lzma converted to acpi.elf and passed to HLOS.
Below is the structure of acpi file for linux.

RSDP  (Root System Description Pointer)
  └── XSDT  (Extended System Description Table)
        ├── FADT  (Fixed ACPI Description Table)  →  DSDT, FACS
        ├── MADT  (Multiple APIC Description Table)
        ├── GTDT  (Generic Timer Description Table)
        ├── IORT  (IO Remapping Table)
        ├── CSRT  (Core System Resource Table)
        ├── PPTT  (Processor Properties Topology Table)
        ├── SPCR  (Serial Port Console Redirection Table)
        ├── DBG2  (Debug Port Table 2)
        └── SSDTs (Secondary System Description Tables) — optional/dynamic

RSDP is the single anchor the OS searches for. On ARM64 EFI, it is placed in the
EFI System Table's Configuration Table (GUID EFI_ACPI_20_TABLE_GUID).

Currently Glymur uses ACPI Version 6.3:

FieldValueMeaning
FADT revision6ACPI 5.0+ family
FADT minor version3→ ACPI 6.3
RSDPNot found in ELF segmentPlaced by UEFI at runtime
HW_REDUCED_ACPI (FADT bit 20)trueNo legacy PM1a/PM1b IO registers — ARM64 mandatory
LOW_POWER_S0_IDLE (FADT bit 21)trueModern standby capable
DSDT AML revision264-bit AML (ACPI 2.0+ integer width)

MADT revision5GICv3/v4 entries supported

2.2 Table Descriptions

RSDP — Root System Description Pointer

The entry point. Firmware places it in EFI System Table on UEFI platforms.

Signature: "RSD PTR "  (8 bytes, no null terminator)
Fields: OEMID, revision, RSDT address (ACPI 1.0), XSDT address (ACPI 2.0+)

Linux finds it via:

// arch/x86/kernel/setup.c  /  arch/arm64/kernel/acpi.c
efi_get_rsdp_addr() → acpi_os_get_root_pointer()

XSDT — Extended System Description Table

Points to all other top-level tables. Array of 64-bit physical addresses.

struct acpi_table_xsdt {
    struct acpi_table_header header;   // Signature "XSDT"
    u64 table_offset_entry[1];         // array of pointers to other tables
};

FADT — Fixed ACPI Description Table

Describes fixed hardware: power management registers, reset register, sleep control.

Key fields for ARM64 HW-Reduced ACPI:

Flags:
  HW_REDUCED_ACPI (bit 20) = 1   → No legacy PM1a/PM1b IO registers
                                    ARM64 always sets this
  LOW_POWER_S0_IDLE (bit 21)     → Modern standby capable
DSDT pointer: 8-byte physical address of DSDT

DSDT — Differentiated System Description Table

The main device namespace. Written in ACPI Source Language (ASL), compiled to
AML bytecode. Contains all Device() objects under \_SB (System Bus).

DefinitionBlock ("dsdt.aml", "DSDT", 2, "QCOM", "GLYMUR", 0x00000001) {
    Scope (\_SB) {
        Device (COM0) {                      // UART device
            Name (_HID, "QCOM0E02")          // Hardware ID — driver match key
            Name (_UID, 0)                   // Unique ID
            Name (_STA, 0xF)                 // Status: present + enabled
            Name (_CRS, ResourceTemplate() {
                Memory32Fixed(ReadWrite,
                    0x00894000,              // MMIO base
                    0x00004000)              // Size
                Interrupt(ResourceConsumer, Level, ActiveHigh, Exclusive) { 0x26B }
            })
        }
    }
}

Key DSDT objects:

Object Purpose
_HID Primary hardware ID (PNP or ACPI ID string) — driver binding
_CID Compatible ID list — secondary match
_UID Unique instance ID
_CRS Current Resource Settings — MMIO ranges, IRQs, DMA
_STA Device status bitmask (present / enabled / shown in UI / functional)
_DSM Device Specific Method — arbitrary firmware RPC
_DSD Device Specific Data — key/value property bag (replaces DT properties)
_PS0/_PS3 Power state methods: D0 (full on) / D3 (off)
_CPC Collaborative Processor Performance Control — CPU freq registers
_PXM Proximity domain — NUMA topology

MADT — Multiple APIC Description Table

CPU and interrupt controller topology.

Entry types on ARM64:
  Type 0x0B: GIC CPU interface entry — one per CPU
             Fields: CPU interface number, MPIDR (affinity), GICC base PA
  Type 0x0C: GIC Distributor (GICD) — one entry
             Fields: physical base address, GIC version
  Type 0x0E: GIC Redistributor (GICR) — describes GICR range
  Type 0x0F: GIC ITS (Interrupt Translation Service) — for MSI

Linux use: early_acpi_boot_init() reads MADT to:
- Count CPUs → pass to smp_init()
- Get GIC base addresses → register GIC driver
- Get timer interrupt lines → arch_timer_acpi_init()

GTDT — Generic Timer Description Table

ARM Generic Timer (arch timer) configuration.

Fields:
  CounterControlBaseAddress  — CNTCTL physical address
  SecurePL1Timer interrupt   — EL1 secure timer IRQ
  NonSecurePL1Timer          — EL1 NS timer IRQ (most kernels use this)
  VirtualTimer               — Virtual timer IRQ
  NonSecurePL2Timer          — Hypervisor timer IRQ
  Platform timers[]          — SBSA GTDT extension: memory-mapped timers

Linux: acpi_arch_timer_init() reads GTDT, registers arch_timer platform device.

IORT — IO Remapping Table

Describes the SMMU (ARM IOMMU) topology and DMA address mappings.

Node types:
  ITS group      — GIC ITS nodes (MSI routing)
  Named component — named DMA master (e.g., PCIe, USB, UFS)
  SMMU v1/v2     — Qualcomm SMMU-500 (SMMUv2)
  SMMU v3        — ARM SMMUv3
  Root complex   — PCIe root complex entry
  Memory range   — DMA window description

ID mappings (SMMU stream IDs):
  Each Named Component has ID mapping array:
  [input_base, id_count, output_base, output_reference]
  Tells SMMU driver which stream IDs belong to which SMMU

Linux: drivers/acpi/arm64/iort.c parses IORT, registers IOMMU groups,
sets up DMA ops for each device.

CSRT — Core System Resource Table

Describes DMA controllers and timer resources that are not in MADT/GTDT.

Resource group:
  Shared info header (vendor ID, device ID, subsystem ID)
  Resource descriptors:
    Type 0: DMA channel
    Type 1: Timer

On Qualcomm: describes BAM DMA channels for QUPV3 if needed.

PPTT — Processor Properties Topology Table

Describes CPU cache topology and cluster hierarchy for cache coherency and
power management.

Node types:
  Processor node — cluster or core
  Cache          — L1/L2/L3 cache properties (size, ways, line size)

Example (SM8480 — 3-cluster):
  Cluster 0 (Silver — 4× Cortex-A520)
    Core 0  → L1D, L1I, L2 (shared cluster)
    Core 1
    ...
  Cluster 1 (Gold — 3× Cortex-A720)
    ...
  Cluster 2 (Prime — 1× Cortex-X4)
    ...
  L3 cache (system cache, shared all clusters)

Linux uses PPTT for cpu_topology and cacheinfo.

SPCR — Serial Port Console Redirection Table

Describes the primary serial console port. Allows OS to configure a serial
console without command-line parameters.

Fields:
  InterfaceType: 0x12 = ARM PL011  /  0x03 = 16550
  BaseAddress: GAS (Generic Address Structure) — MMIO base
  InterruptType: GIC (arm64)
  Interrupt: GSI number
  BaudRate: encoded (0x07 = 115200)
  FlowControl: none / HW

Linux: acpi_parse_spcr() → registers earlycon if no earlycon= on cmdline.

DBG2 — Debug Port Table 2

Describes additional debug UART(s) for OS debugger use (e.g., KDGB serial).

SSDT — Secondary System Description Table

Optional tables that extend the DSDT namespace. Can be:
- Static: loaded by firmware at boot alongside DSDT
- Dynamic (initramfs overlay): injected by CONFIG_ACPI_TABLE_UPGRADE
— Linux scans uncompressed cpio prepend for kernel/firmware/acpi/*.aml

Example use: adding a device not present in base DSDT (e.g., uart21_ssdt.aml
adds \_SB.QURC for UART21 on Glymur).

4. ACPI Namespace and Driver Binding

4.1 Namespace Structure

The ACPI namespace is a tree of named objects built from DSDT + SSDT bytecode.
Device objects live under \_SB (System Bus).

\
└── _SB  (System Bus — root for all hardware)
    ├── PCI0  (PCIe root complex)
    ├── GIO0  (GPIO controller)
    ├── I2C0  (I2C bus)
    ├── COM0  (UART — _HID "QCOM0E02")
    ├── UFS0  (UFS storage)
    ├── USB0  (USB controller)
    └── CPU0..CPUN  (_HID "ACPI0007", _CPC for cpufreq)

4.2 Driver Binding Chain

DSDT Device() with _HID = "QCOM0E02"
        │
        ▼
acpi_scan_init()
  → struct acpi_device created
  → bus_type = acpi_bus_type

        │
        ▼
acpi_apd.c (ACPI Platform Driver — generic layer)
  → acpi_apd_match_table[] entry for "QCOM0E02":
      .mode = APD_MODE_CLOCK
      .fixed_clk_rate = 7372800  (7.3728 MHz)
      .clk_conn_names = {"se", NULL}
  → acpi_apd_setup() injects clock metadata
  → platform_device_alloc() + platform_device_add()

        │
        ▼
platform bus matches driver (e.g., qcom_geni_serial)
  → driver probe() called
  → has_acpi_companion(dev) == true
  → uses _CRS for MMIO + IRQ
  → uses fixed_clk_rate (no clk_get())

4.3 Resource Acquisition in ACPI Mode

// MMIO — from _CRS Memory32Fixed()
ret = platform_get_resource(pdev, IORESOURCE_MEM, 0);
base = devm_ioremap_resource(dev, ret);

// IRQ — from _CRS Interrupt()
irq = platform_get_irq(pdev, 0);

// Clock — from APD fixed_clk_rate (NO clock framework call)
clk_rate = dev_pm_opp_get_freq(dev);  // or read from platform_data
// NOT: devm_clk_get(dev, "se")  ← this fails in ACPI mode (no clock framework)

// Power domain — from _PS0/_PS3 methods (via ACPI PM)
// NOT: devm_pd_get()  ← DT-only

6. Glymur ACPI Bringup — Issues Encountered and Fixes

This section documents every issue hit while bringing up ACPI boot on Glymur (SM8480)
and the corresponding fix or workaround.

6.1 UEFI Issues

Issue A — SPSS/TME/MCP Auth Failure → SMMU Unidentified Stream Fault

Root cause:

TME loads MCP firmware and brings MCP out of reset before TZ authentication
completes (standard PIL flow). TZ then fails authentication
(TzStatus = 0xFFFFFFFE, BIST_CTRL fuse unprogrammed) and returns an error.
However, TZ only verifies the image — it does not re-assert MCP's reset or
cut its power after a failed auth. TME does not handle the auth failure cleanly,
so MCP stays running.

MCP firmware proceeds with its normal init and issues DMA transactions
(StreamID 0x1c001de0). In UEFI this is harmless — the SMMU is in bypass mode.
Once Linux enables SMMU enforcement, MCP's DMA hits an Unidentified Stream
Fault because no SMMU context was ever configured (TZ auth failed before that
step, so stream-ID programming was never done).

Failure sequence:

SPSS PIL load → TME loads MCP → TZ auth fails (BIST_CTRL fuse unprogrammed)
  └─ TZ does NOT re-assert MCP reset
       └─ MCP runs, issues DMA (StreamID 0x1c001de0)
            └─ UEFI: SMMU bypass → harmless
                 └─ Linux enables SMMU enforcement
                      └─ MCP DMA → SMMU USF crash → kernel panic

Fix: Set PcdDisableSPSSBoot = TRUE in spss.dtsi (UEFI source).
This causes SPSSLib to return EFI_LOAD_ERROR early, preventing SPSS/TME/MCP
from being loaded at all.

Reference patch: UEFI_SPSS_disable_changes.patch

Workaround (if UEFI change not available): Add arm-smmu.disable_bypass=0
to the kernel command line. This keeps the SMMU in bypass mode so MCP's
unauthenticated DMA does not fault.

Note: The workaround disables SMMU enforcement globally and should only
be used for bringup/debug. The proper fix is the UEFI PcdDisableSPSSBoot change.

Issue B — UEFI Variable / Boot Path Selection

To select the ACPI boot path, one of the following must be done:

Option 1 — Set UEFI variable:

OsConfigTableSelection = 0x0

This causes DtPlatformDxe to select DT_ACPI_SELECT_ACPI and install the
RSDP into the EFI Configuration Table. Also set boot order to "Qclinux".

Option 2 — Run NSH script in UEFI shell:

Boot_Linux_ACPI.nsh

Without one of these, DtPlatformDxe defaults to DT mode and Linux never finds
the RSDP — ACPI init does not start.

6.2 Kernel Config

Issue — ACPI Namespace Load Fails Without CONFIG_ACPI_REDUCED_HARDWARE_ONLY

CONFIG_ACPI_REDUCED_HARDWARE_ONLY=y

Qualcomm ACPI tables already set HW_REDUCED_ACPI in the FADT flags. This flag
means legacy PC-style PM1a/PM1b IO registers do not exist. Without this Kconfig
option, the kernel still attempts to access legacy hardware regions during ACPI
init and fails.

Kconfig dependency added:

config ACPI_REDUCED_HARDWARE_ONLY
    bool
    select ACPI_REDUCED_HARDWARE_ONLY if ACPI

6.3 Kernel Driver Changes

Reference patch: Kernel_my_changes.patch

Fix A — drivers/soc/qcom/qcom-geni-se.c

Problem: geni_se_get_qup_hw_version() dereferences se->wrapper to read
the QUP hardware version register. In DT mode, se->wrapper points to the QUP
wrapper parent device. In ACPI mode, the device hierarchy is flat — there is no
QUP wrapper parent device, so se->wrapper == NULL → NULL pointer dereference.

Fix: Add a NULL guard that returns a hardcoded version instead of
dereferencing:

// if se->wrapper is NULL (ACPI flat hierarchy), return SM8480 QUP version
if (!se->wrapper)
    return 0x300A0000;   // QUP v3.10 for SM8480

Fix B — drivers/tty/serial/qcom_geni_serial.c (3 fixes)

Fix B1 — request_port(): skip re-map if already mapped

In ACPI mode, qcom_geni_serial_probe() must pre-map MMIO registers via
request_port() before calling uart_add_one_port() (the UART core calls
request_port() again internally). Without a guard, the second call
re-maps the same region and overwrites uport->membase.

// Skip ioremap if registers already mapped (ACPI pre-map path)
if (uport->membase)
    return 0;

Fix B2 — geni_serial_resource_init(): early return in ACPI mode

In DT mode this function calls devm_clk_get(dev, "se") and icc_get().
Both fail in ACPI mode — the clock framework and ICC framework are DT-only.
Firmware (XBL/UEFI) already set the UART clock frequency and bus config.

// In ACPI mode: firmware owns clock + bus. Skip DT-only resource init.
if (has_acpi_companion(dev))
    return 0;

Fix B3 — qcom_geni_serial_probe(): pre-map registers before uart_add_one_port()

In ACPI mode, uart_add_one_port() triggers request_port() which needs
uport->membase already set. Pre-map via request_port() in probe before
the uart_add_one_port() call.

if (has_acpi_companion(&pdev->dev)) {
    ret = uport->ops->request_port(uport);
    if (ret)
        return ret;
}

Fix C — drivers/acpi/acpica/tbxfload.c

Problem: acpi_ev_install_region_handlers() returns AE_BAD_PARAMETER
when registering handlers for SystemIO and other legacy hardware regions.
On HW-Reduced ACPI (ARM64), these regions do not exist — the error is expected.
However, ACPICA treated this as fatal and aborted the namespace load, preventing
acpi_tb_load_namespace() from completing.

Fix: Make AE_BAD_PARAMETER non-fatal during region handler installation:

// AE_BAD_PARAMETER is expected on HW_REDUCED_ACPI platforms
// (legacy IO regions absent) — treat as non-fatal and continue
if (status == AE_BAD_PARAMETER)
    status = AE_OK;

Without this fix, ACPI namespace load fails entirely and no devices are
discovered.

6.4 Ramdisk — UART21 SSDT Injection (Workaround)

Problem: The base ACPI tables (acpi.elf on BOOT_FW2) do not include a
Device node for UART21. qcom_geni_serial has no ACPI device to bind to and
/dev/ttyMSM0 is never registered — no console.

Workaround: Prepend an uncompressed cpio archive containing the UART21 SSDT
AML file before the main ramdisk. CONFIG_ACPI_TABLE_UPGRADE causes Linux to
scan the uncompressed cpio prepend for kernel/firmware/acpi/*.aml and inject
them into the ACPI namespace.

Ramdisk layout:
  [uncompressed cpio] kernel/firmware/acpi/uart21.aml
  [main ramdisk.gz]

Result:
  acpi_table_upgrade() loads uart21.aml
  → \_SB.QURC (Device, _HID = "QCOM0E02") appears in namespace
  → acpi_apd matches → platform_device created
  → qcom_geni_serial probe() → /dev/ttyMSM0 registered

SSDT provides:
- MMIO base: 0x894000 (size 0x4000)
- IRQ: 619 (0x26B)
- _HID = "QCOM0E02" (matched by qcom_geni_serial acpi_match_table)

compilation procedure:
you need to create peripheral dsl file as per schematic and property (uart21_ssdt.dsl)
iasl uart21_ssdt.dsl -> uart21_ssdt.aml --> compile dsl file and generate aml.
keep the generated aml file as cpio format in /kernel/firmware/acpi -> this will be picked during kernel boot and append to the existing name space.

To prepare run time SSDT Table in ramdisk

mkdir -p kernel/firmware/acpi
cp uart21_ssdt.aml kernel/firmware/acpi/
find kernel/firmware/acpi/uart21_ssdt.aml | cpio -o --format=newc > ssdt_prepend.cpio   # NO -v flag!

# Prepend before the main gzip ramdisk
cat ssdt_prepend.cpio original_ramdisk.cpio.gz > new_ramdisk.cpio.gz

Long-term fix: Include the UART21 SSDT in acpi.elf on the BOOT_FW2
partition so it is loaded directly by AcpiPlatformDxe at UEFI boot — no
ramdisk overlay needed.

11. Limitaions of Full ACPI on Qualcomm SoCs

  1. SCMI Server Not Implemented in AOSS Firmware (Root Blocker)

The most fundamental gap. SCMI is the ARM-standard replacement for all DT-based resource management in ACPI mode.

Two-sided failure:
- UEFI side: No _HID "ARMH0011" SCMI device node in DSDT. Linux arm_scmi driver has nothing to bind to.
- AOSS firmware side: No SCMI server dispatch loop. AOSS currently uses Qualcomm-proprietary QMP protocol — not SCMI.

Consequence: cpufreq-scmi, scmi-devfreq, clk-scmi consumers all fail at probe. No performance/clock/power domain handles registered.

  1. ICC (Interconnect / DDR Bandwidth Voting) — DT-Only, No ACPI Backend

icc_get() internally calls of_icc_get(). There is no acpi_icc_get() in upstream kernel. drivers/interconnect/qcom/icc-rpmh.c uses rpmh_write() via DT path only.

Consequence: DDR bandwidth voting silently disabled in ACPI mode. GPU, camera, USB, CPU all lose DDR frequency control. DDR runs at whatever AOSS's autonomous activity monitor picks — no
Linux visibility.

  1. OPP (Operating Performance Points) — DT-Only, No ACPI Equivalent

OPP tables (operating-points-v2) are DT properties. dev_pm_opp_of_add_table() has no ACPI counterpart. The required-opps → power-domains linkage (which ties clock level to VDD_CX vote)
has no ACPI representation.

Consequence: GPU devfreq broken. No {freq → voltage} table → devfreq governor has no frequency steps. GPU runs at XBL/UEFI boot frequency permanently. No thermal-aware GPU scaling.

  1. GCC Peripheral Clock Drivers — DT-Only

All Qualcomm GCC clock drivers (clk-gcc-sm8550.c, etc.) use of_match_table and bind to DT nodes. No ACPI match table or _HID defined. Clock registration never happens in ACPI mode.

Consequence: Peripheral clocks for I2C, SPI, USB, camera, display cannot be programmed by Linux. Drivers that call devm_clk_get() fail probe. Workaround today: skip clk_get() in ACPI
mode and rely on firmware-set rates (works only for simple fixed-rate peripherals like UART).

  1. QMP Mailbox (AOSS Communication) — DT-Only

qcom_aoss.c and qcom-apcs-ipc-mailbox.c bind via of_platform_populate(). In ACPI mode of_platform_populate() never runs → QMP mailbox never registers → no Linux→AOSS communication
channel at all.

Consequence: System sleep coordination (DDR collapse, voltage corner transitions) loses its Linux-side path. AOSS operates fully autonomously — Linux cannot request or observe power
state changes.

  1. Power Domain / Regulator Framework — DT-Only (for RPMh-backed regulators)

qcom-rpmhpd.c registers RPMh power domains via DT power-domains phandles. In ACPI mode, dev_pm_opp_set_rate() cannot find the genpd power domain → voltage scaling is skipped even if a
frequency switch is attempted.

Consequence: Clock scaling without voltage scaling → potential timing violations or running below required voltage margin.

  1. Complex Hierarchical Device Topology Flattened

DT expresses QUP → SE hierarchy, clock parents, power domain graphs. ACPI DSDT represents flat platform devices. Drivers that assume se->wrapper is non-NULL (parent QUP wrapper device)
crash.

Consequence: Every subsystem driver that traverses parent-child device hierarchy needs an explicit ACPI guard. This is not one fix — it is a sustained per-driver porting effort.

  1. No ACPI Support for Qualcomm-Specific Subsystems (PIL, remoteproc, etc.)

Peripheral image loading (qcom_pil_info.c, remoteproc) for ADSP, CDSP, MPSS is entirely DT-based. ACPI has no equivalent of the qcom,pil-* compatible binding chain.

Consequence: DSP subsystems (audio, compute, modem) cannot be loaded or managed by Linux in ACPI mode. UEFI handles PIL at boot but dynamic load/unload is impossible.

  1. Thermal Management — Partial

qcom-spmi-temp-alarm.c, tsens.c bind via DT. In ACPI mode thermal zone definitions (_TZ, _TSP) would be needed in DSDT plus SCMI sensor protocol for AOSS-reported temperatures. Neither
is present.

Consequence: Only basic ACPI thermal zones (if defined in DSDT) work. No Qualcomm-specific junction temperature sensors → no fine-grained thermal throttling.
